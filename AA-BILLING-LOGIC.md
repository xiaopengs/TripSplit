# AA 记账小程序 — 多人协作 & 影子成员 完整逻辑规范

> 本文档面向开发人员，确保所有边界场景和状态逻辑不遗漏。

---

## 一、核心概念定义

| 概念 | 说明 |
|------|------|
| **账本** | 一个独立的记账空间，如「日本旅行」「日常饭局」 |
| **创建者** | 创建账本的人，拥有管理权限（添加成员、删除账本等） |
| **真实成员** | 已通过微信登录、加入账本的用户 |
| **影子成员** | 创建者预先添加的虚拟占位成员，尚未绑定真实微信用户 |
| **认领** | 真实用户绑定一个影子成员，"影子"变为"真人" |
| **记账记录** | 一笔消费：谁付的钱、参与了谁、金额多少 |
| **结算** | 计算每个人该出/该收多少，生成待结算清单 |

## 二、成员模型 & 状态机

### 2.1 成员状态

```
影子成员(未认领) ──认领──→ 真实成员(已绑定)
     │                        │
     │ (可被删除)              │ (可退出账本)
     ↓                        ↓
    已删除                   已退出
```

### 2.2 成员数据结构（关键字段）

```typescript
interface Member {
  id: string;              // 成员唯一ID
  ledgerId: string;        // 所属账本ID
  displayName: string;     // 显示名称（创建者起的昵称，如"老王"）
  type: 'shadow' | 'real'; // 影子 or 真实
  userId?: string;         // 绑定的微信用户ID（认领后才有）
  avatarUrl?: string;      // 头像（认领后取微信头像，未认领取默认）
  createdAt: number;       // 加入时间
  claimedAt?: number;      // 认领时间
}
```

### 2.3 ⚠️ 关键原则

> **`displayName` 是账本内的唯一标识名称，认领后不变。**
> 
> 创建者添加影子成员时起名"老王"，老王认领后，所有人看到的都是"老王"，不会变成"创建者"或"用户XXX"。

## 三、认领流程（核心！）

### 3.1 完整流程

```
创建者A: 添加影子成员"老王"、"小李"
创建者A: 记一笔 — A付了300，参与人：A、老王、小李
创建者A: 分享账本链接给微信好友B

用户B: 打开链接 → 看到未认领的影子成员列表 → 选择"老王" → 认领

此时:
  - 老王.type = 'real', 老王.userId = B.userId
  - 老王.displayName 不变，仍是"老王"
  - A 的界面：老王旁边出现"已认领"标记
  - B 的界面：自己在账本中的名称是"老王"
```

### 3.2 认领后的身份映射

```
用户B（微信用户） ←→ 老王（账本成员）
```

- B 在自己界面上，看到"老王"旁边有"(我)"标记
- A 在自己界面上，看到"老王"旁边有"已认领"标记
- **B 记一笔账，显示的是"老王记了一笔"，不是"用户B记了一笔"**

### 3.3 认领约束

| 规则 | 说明 |
|------|------|
| 一个影子只能被一个用户认领 | 防止多人认领同一个影子 |
| 一个用户在一个账本中只能认领一个影子 | 一个账本一个人 |
| 已认领的影子不能被其他人再次认领 | 并发认领需加锁/校验 |
| 创建者不需要认领 | 创建者自动是真实成员 |

### 3.4 分享链接逻辑

```
分享链接 = 小程序页面路径 + ledgerId + inviteToken

打开链接的用户:
  1. 未登录 → 引导登录
  2. 已登录 + 已在该账本 → 直接进入账本
  3. 已登录 + 不在该账本 → 显示未认领的影子列表，选择认领
  4. 已登录 + 该账本没有未认领影子 → 提示"该账本成员已满"
```

## 四、记账记录逻辑

### 4.1 一笔记录的数据结构

```typescript
interface BillRecord {
  id: string;               // 记录唯一ID
  ledgerId: string;         // 账本ID
  payerId: string;          // 付款人 memberId（不是 userId！）
  amount: number;           // 总金额（单位：分，避免浮点问题）
  currency: string;         // 币种
  description: string;     // 描述，如"午餐"
  participantIds: string[]; // 参与分摊的 memberId 列表
  splitType: 'equal' | 'custom' | 'percentage'; // 分摊方式
  customAmounts?: Record<string, number>; // 自定义分摊金额 {memberId: amount}
  createdBy: string;        // 创建者 memberId
  createdAt: number;        // 创建时间
  updatedAt: number;        // 更新时间
  deletedAt?: number;       // 软删除时间
}
```

### 4.2 ⚠️ 核心原则：统一用 `memberId`，不用 `userId`

> **这是你当前 bug 的根源！**
>
> 所有记录关联的都是 `memberId`（即"老王"），不是 `userId`（即"用户B的微信ID"）。
>
> 渲染界面时：根据 `memberId` 查 `Member` 表，取 `displayName` 显示。
> - `payerId = "member_003"` → 查 Member 表 → `displayName = "老王"` → 显示"老王付了 ¥300"
> - **绝对不能显示"创建者付了 ¥300"**，因为"创建者"是角色，不是名字

### 4.3 不同用户的界面展示规则

以一笔记录为例：**老王付了 ¥300，参与人：A、老王、小李**

| 查看者 | 付款人显示 | 自己的角色 | 分摊金额 |
|--------|-----------|-----------|---------|
| A（创建者） | 老王 | 参与人 | ¥100 |
| B（认领了老王） | **我（老王）** | 付款人 | ¥100 |
| C（认领了小李） | 老王 | 参与人 | ¥100 |
| D（无关用户） | 不可见此账本 | - | - |

**展示规则：**
- 如果 `payerId === 当前用户的 memberId` → 显示 **"我"** 或 **"我（老王）"**
- 如果 `payerId !== 当前用户的 memberId` → 显示该成员的 `displayName`
- **任何情况下都不显示"创建者"这个角色词**

### 4.4 记录同步机制

```
A 记了一笔 → 写入数据库 → 推送通知给其他在线成员 → 其他成员拉取最新记录

同步规则:
  1. 所有记录存在服务端，客户端不缓存为权威数据源
  2. 进入账本页面时，拉取全量记录
  3. 新增/修改/删除记录时，实时推送（WebSocket / 轮询）
  4. 每条记录有唯一 ID，客户端按 ID 去重，**绝不允许同一条记录显示两次**
  5. 离线时本地缓存操作，上线后以服务端数据为准合并
```

### 4.5 ⚠️ 去重规则

```typescript
// 拉取记录后，按 record.id 去重
const uniqueRecords = new Map<string, BillRecord>();
records.forEach(r => uniqueRecords.set(r.id, r));
// 渲染 uniqueRecords.values()
```

> **绝对不能用时间戳或描述去重**，必须用服务端生成的唯一 ID。

## 五、结算逻辑

### 5.1 计算模型

```
每人应付 = 该人参与的所有记录中，其分摊金额之和
每人已付 = 该人作为 payer 的所有记录金额之和
每人余额 = 已付 - 应付

余额 > 0 → 该人应该收回钱（债主）
余额 < 0 → 该人应该付钱（欠债人）
```

### 5.2 结算算法（最小交易数）

```typescript
function calculateSettlements(members: Map<string, number>): Settlement[] {
  // members: { memberId: balance }
  const creditors: [string, number][] = [];  // 正余额
  const debtors: [string, number][] = [];    // 负余额
  
  for (const [id, balance] of members) {
    if (balance > 0.01) creditors.push([id, balance]);
    else if (balance < -0.01) debtors.push([id, -balance]);
  }
  
  creditors.sort((a, b) => b[1] - a[1]);
  debtors.sort((a, b] => b[1] - a[1]);
  
  const settlements: Settlement[] = [];
  let i = 0, j = 0;
  
  while (i < debtors.length && j < creditors.length) {
    const amount = Math.min(debtors[i][1], creditors[j][1]);
    if (amount > 0.01) {
      settlements.push({
        from: debtors[i][0],   // 欠债人 memberId
        to: creditors[j][0],    // 债主 memberId
        amount: Math.round(amount * 100) / 100  // 精确到分
      });
    }
    debtors[i][1] -= amount;
    creditors[j][1] -= amount;
    if (debtors[i][1] < 0.01) i++;
    if (creditors[j][1] < 0.01) j++;
  }
  
  return settlements;
}
```

### 5.3 结算展示

```
结算清单:

小李 需要付给 老王: ¥150
小李 需要付给 A:    ¥50
```

**展示规则：**
- `from` 成员名 → 显示 `displayName`
- `to` 成员名 → 显示 `displayName`
- 当前用户如果是 `from` → "你需要付给 老王 ¥150"
- 当前用户如果是 `to` → "小李需要付给你 ¥150"

### 5.4 结算状态

```typescript
interface Settlement {
  id: string;
  ledgerId: string;
  fromMemberId: string;     // 付款方 memberId
  toMemberId: string;       // 收款方 memberId
  amount: number;           // 结算金额
  status: 'pending' | 'confirmed_by_from' | 'confirmed_by_to' | 'settled';
  settledAt?: number;
}
```

| 状态 | 含义 |
|------|------|
| `pending` | 待结算 |
| `confirmed_by_from` | 付款方确认已付 |
| `confirmed_by_to` | 收款方确认已收 |
| `settled` | 双方都确认，结算完成 |

> **必须双方确认才能标记为已结算**，防止一方说付了另一方说没收到。

## 七、边界场景 & 异常处理

### 7.1 并发场景

| 场景 | 处理方案 |
|------|---------|
| 两人同时认领同一影子 | 服务端校验，后者提示"已被认领"，刷新列表 |
| 两人同时记一笔 | 各自独立生成 recordId，不冲突 |
| 记账同时有人在结算 | 结算基于快照，记账不影响进行中的结算 |
| 网络断开时记账 | 本地缓存，上线后提交，若冲突提示用户 |

### 7.2 成员变动场景

| 场景 | 处理方案 |
|------|---------|
| 影子成员被删除，但已有记录引用 | 记录保留，payerId/displayName 不变（冻结），标记该成员为 deleted |
| 真实成员退出账本 | 历史记录保留，结算仍需完成，新记录不再包含该成员 |
| 退出后又想回来 | 需创建者重新添加影子并认领（或直接邀请） |
| 创建者退出 | 不允许，只能归档或删除账本 |

### 7.3 金额精度

| 规则 | 说明 |
|------|------|
| 所有金额以 **分** 为单位存储 | 避免浮点误差 |
| 前端展示时除以100 | 显示为元 |
| 分摊时无法整除 | 余数加到最后一个分摊人（或创建者），并在记录中标注 |
| 币种转换 | 如有多币种需求，记录原始币种+汇率，结算时统一换算 |

### 7.4 其他边界

| 场景 | 处理方案 |
|------|---------|
| 一笔记录只有付款人参与 | 允许（比如替人代付，分摊人为空则全记在付款人头上） |
| 金额为 0 | 允许，用于记录"请客"或备注 |
| 修改已有记录 | 记录修改历史，结算重新计算 |
| 删除记录 | 软删除（deletedAt），保留审计轨迹，结算时排除 |

## 九、数据同步架构建议

```
┌────────┐     ┌────────┐     ┌────────┐
│  客户端A │────▶│  服务端  │◀────│  客户端B │
│ (创建者) │◀────│        │────▶│ (老王)  │
└────────┘     └────────┘     └────────┘
                  │
                  ▼
             ┌────────┐
             │ 数据库  │
             │        │
             │ - ledgers   (账本)
             │ - members   (成员)
             │ - records   (记录)
             │ - settlements(结算)
             └────────┘
```

### 同步策略

1. **权威数据源在服务端** — 客户端所有操作都走 API
2. **实时推送** — WebSocket 或小程序云开发实时数据监听
3. **乐观更新** — 本地先更新 UI，API 返回后校验，失败则回滚
4. **全量拉取 + 增量推送** — 进入页面拉全量，之后监听增量变化
5. **冲突解决** — 以服务端时间戳为准，后到者覆盖

### API 设计要点

```
GET    /api/ledgers/:id                    # 获取账本信息
GET    /api/ledgers/:id/members            # 获取成员列表
GET    /api/ledgers/:id/records            # 获取记录列表（分页）
POST   /api/ledgers/:id/records            # 新增记录
PUT    /api/ledgers/:id/records/:rid       # 修改记录
DELETE /api/ledgers/:id/records/:rid       # 删除记录（软删除）
POST   /api/ledgers/:id/claim              # 认领影子成员
POST   /api/ledgers/:id/settle             # 发起结算
PUT    /api/ledgers/:id/settlements/:sid   # 确认结算
```

## 十、当前 Bug 定位与修复指引

### Bug 1: "创建者付款" 而非显示具体名字

**根因：** 前端渲染时用了角色（creator）而非 `member.displayName`

**修复：**
```javascript
// ❌ 错误写法
const payerName = record.payerId === currentUserId ? '我' : 
                 record.payerId === ledger.creatorUserId ? '创建者' : '成员';

// ✅ 正确写法
const payerMember = membersMap.get(record.payerId);
const payerName = record.payerId === myMemberId ? `我（${payerMember.displayName}）` 
                 : payerMember.displayName;
```

### Bug 2: A 的界面出现重复记录

**根因：** 缺少按 `record.id` 去重，可能拉取了多次数据直接 concat

**修复：**
```javascript
// ❌ 错误写法
records = [...oldRecords, ...newRecords];  // 可能重复

// ✅ 正确写法
const map = new Map();
[...oldRecords, ...newRecords].forEach(r => map.set(r.id, r));
records = Array.from(map.values());
```

### Bug 3: B 记的账 A 看不到

**根因：** 记录存储时用了 `userId` 而非 `memberId`，或同步机制缺失

**修复：**
- 确认 `records` 表存的是 `payerId = memberId`
- 确认 A 进入账本时拉取全量 records
- 确认 B 新增记录后触发推送通知
